# Admiral Framework v0.2.0-alpha — Architecture Decisions

> These items surfaced during the v0.1.x audit. Each requires a design decision
> before implementation. Three candidate solutions are listed per item; the
> recommended option is marked **[Recommended]**.

---

## 1. Orchestrator Single Point of Failure / Failover

**Current state:** The Orchestrator (Tier 1) is the sole coordinator for task
decomposition, routing, and progress tracking. Governance agents can bypass a
degraded Orchestrator and escalate directly to the Admiral, but there is no
proactive failover mechanism.

### Option A — Hot Standby Orchestrator

Run a second Orchestrator instance that mirrors the primary's decision log and
task state. On heartbeat failure (3 consecutive misses over 30 s), the standby
promotes itself and the governance layer re-routes.

- **Pro:** Near-zero downtime; transparent to specialists.
- **Con:** Doubles Tier 1 token spend during normal operation; state
  synchronization adds complexity and potential for split-brain.

### Option B — Admiral-as-Fallback Decomposer **[Recommended]**

Extend the Admiral to perform coarse-grained task decomposition when the
Orchestrator is unresponsive. Governance agents already escalate to the Admiral
on degradation. Formalize this path: the Admiral decomposes the current goal
into 1-3 macro-tasks and routes directly to Tier 1 specialists until the
Orchestrator recovers.

- **Pro:** No idle standby cost; leverages the existing governance bypass path;
  the Admiral already has full fleet context. Minimal new infrastructure.
- **Con:** Admiral decomposition is coarser — fine-grained parallelism degrades
  during the outage window; Admiral context window takes on additional load.

**Why this is best:** It builds on the escalation path that already exists in
`governance.md`, requires no new agents or synchronization protocol, and
accepts a graceful degradation (coarser routing) rather than paying the
continuous cost of a hot standby.

### Option C — Peer-Elected Orchestrator

Allow any Tier 1 agent to be elected as temporary Orchestrator via a simple
quorum vote among active agents. The elected agent inherits the last
checkpointed task state.

- **Pro:** No dedicated standby; distributes responsibility.
- **Con:** Election protocol is complex for LLM agents; elected agent loses its
  specialist capacity; checkpoint format must be fully specified.

## 2. Identity Authority Single Point of Failure

**Current state:** The Authentication & Identity Specialist designs auth flows,
token lifecycle, and OAuth/OIDC federation. There is no documented fallback if
the specialist is unavailable, and no separation between identity *design* and
identity *runtime enforcement*.

### Option A — Split Design vs. Runtime Enforcement

Separate the specialist into two concerns: (1) an **Identity Designer** agent
that architects auth flows, and (2) a set of **deterministic hooks** (e.g.,
`PreToolUse` token-validation hook) that enforce identity at runtime without
requiring the specialist to be online.

- **Pro:** Runtime enforcement is deterministic and doesn't depend on an LLM
  agent being available; clear separation of concerns.
- **Con:** Two-component system to maintain; hook logic can't handle novel auth
  patterns without the designer.

### Option B — Redundant Specialist Pool **[Recommended]**

Allow the Identity Specialist to be instantiated as a stateless pool (N >= 2).
Because the specialist's knowledge is entirely prompt-driven (no mutable
state), any instance can handle any request. Combine this with a
`SessionStart` hook that validates the auth configuration artifact exists
before any agent session begins — making the *runtime* identity enforcement
hook-based and agent-independent.

- **Pro:** Stateless agents are trivially scalable; the hook provides a
  deterministic safety net even if no specialist is available; no architecture
  change to existing agent definitions.
- **Con:** Pool management adds orchestration overhead; doesn't help if the
  underlying model API is down (see item 7).

**Why this is best:** It acknowledges that the real SPOF isn't the agent but the
*absence of deterministic runtime enforcement*. The hook handles runtime; the
pool handles design-time availability. Both are incremental additions.

### Option C — Federated Identity Delegation

Delegate identity concerns to the platform layer entirely (e.g., API gateway
enforces auth; agents never touch tokens directly). The specialist only
produces configuration artifacts consumed by the gateway.

- **Pro:** Agents are fully decoupled from identity runtime.
- **Con:** Requires a specific platform architecture; reduces framework
  portability; the specialist becomes a config generator rather than a
  first-class agent.

## 3. Governance Self-Monitoring

**Current state:** Seven governance agents monitor the fleet, with an
Authoritative Ownership Table and conflict resolution. But nothing monitors
the governance agents themselves — a silent failure in the Drift Monitor, for
example, would go undetected.

### Option A — Governance Heartbeat Hook

Add a deterministic `GovernanceHeartbeat` hook that fires on a configurable
interval. Each governance agent must emit a structured health signal. If any
agent misses two consecutive heartbeats, the hook alerts the Admiral directly.

- **Pro:** Simple, deterministic, no new agents needed.
- **Con:** Heartbeat only detects liveness, not correctness (a governance agent
  could be alive but producing bad assessments).

### Option B — Cross-Governance Audit Rotation **[Recommended]**

Implement a rotating audit schedule where each governance agent periodically
reviews one other governance agent's recent outputs. The rotation ensures every
agent is audited by a *different* agent each cycle. Findings route to the
Admiral, not the Orchestrator.

- Rotation example (weekly cycle):
  - Week 1: Token Budgeter audits Drift Monitor; Drift Monitor audits
    Hallucination Auditor; ... (round-robin)
  - Week 2: Rotation shifts by one position.

- **Pro:** Detects both liveness *and* correctness issues; uses existing agent
  capabilities; no new infrastructure.
- **Con:** Adds token overhead for audit tasks; governance agents auditing each
  other could develop blind spots over time (mitigated by rotation).

**Why this is best:** It catches degraded *quality*, not just crashes. A
heartbeat tells you the Bias Sentinel is running; a cross-audit tells you it's
actually catching bias. The rotation prevents collusion-like blind spots. This
is the "who watches the watchmen" answer that matches the framework's existing
peer-review philosophy.

### Option C — Dedicated Meta-Governance Agent

Create a new Tier 1 agent ("Governance Auditor") whose sole purpose is
monitoring the seven governance agents.

- **Pro:** Clear ownership; single point of accountability.
- **Con:** Recursive SPOF problem (who monitors the meta-governor?); adds a new
  Tier 1 agent and its associated token cost; increases fleet complexity.

## 4. Hook Configuration Format Specification

**Current state:** Hooks use a well-defined contract (JSON stdin, exit codes,
timeout) but there is no versioning, no schema registry, and no mechanism for
hooks to declare their capabilities or dependencies.

### Option A — JSON Schema Registry

Define a formal JSON Schema for each hook lifecycle event's input/output
contract. Publish schemas in a `hooks/schemas/` directory. Hooks declare which
schema version they conform to via a `"schema_version"` field. The runtime
validates payloads against schemas before invoking hooks.

- **Pro:** Formal validation; tooling can auto-generate hook stubs; breaking
  changes are version-gated.
- **Con:** Schema maintenance overhead; may over-formalize what is currently a
  simple contract.

### Option B — Hook Manifest File **[Recommended]**

Each hook ships with a `hook.manifest.json` that declares:
```json
{
  "name": "token_budget_gate",
  "version": "1.0.0",
  "events": ["PreToolUse"],
  "timeout_ms": 5000,
  "requires": ["token_budget_tracker"],
  "input_contract": "v1",
  "description": "Blocks tool invocations that would exceed the session token budget."
}
```
The runtime reads manifests at `SessionStart`, validates dependency chains, and
rejects sessions with unsatisfied hook dependencies. Contract versions are
simple strings (`v1`, `v2`) — not full JSON Schemas.

- **Pro:** Lightweight; captures dependencies (critical for hook ordering);
  self-documenting; easy to extend without a schema registry.
- **Con:** Less formal validation than full JSON Schema; relies on convention
  for contract versioning.

**Why this is best:** It solves the three actual problems — dependency ordering,
capability declaration, and version compatibility — without the overhead of a
schema registry. The manifest is the minimum viable specification that enables
tooling and validation. Full JSON Schema can be added later as `v2` if needed.

### Option C — Convention-Only (Status Quo + Documentation)

Document the existing contract more thoroughly. Add naming conventions
(`pre_*.sh`, `post_*.sh`) and a recommended directory structure. No runtime
validation.

- **Pro:** Zero implementation cost; no new moving parts.
- **Con:** Doesn't solve dependency ordering or version compatibility;
  convention drift is likely as the hook ecosystem grows.

## 5. Attack Corpus Bootstrapping

**Current state:** The Chaos Agent injects pre-defined failure scenarios
(network partitions, resource exhaustion, clock skew) but there is no
documented strategy for seeding, growing, or learning from an attack corpus.
The Red Team Agent and Assumption Inversion Agent perform adversarial review
but don't feed findings back into a reusable corpus.

### Option A — Static Corpus with Manual Curation

Create an `attack-corpus/` directory with categorized scenario files (YAML or
Markdown). Scenarios are added manually after incidents or red team exercises.
The Chaos Agent reads from this directory at session start.

- **Pro:** Simple; human-curated quality; no automation risk.
- **Con:** Corpus stagnates without active curation; no feedback loop from
  production incidents.

### Option B — Feedback-Loop Corpus **[Recommended]**

Establish a three-source pipeline that feeds into a shared corpus stored in the
Brain's semantic memory:

1. **Red Team findings** — After each adversarial review, the Red Team Agent
   writes a structured scenario entry (trigger, expected behavior, actual
   behavior, severity).
2. **Incident post-mortems** — The Incident Response Agent extracts failure
   scenarios from post-mortems and writes them as corpus entries.
3. **Chaos Agent discoveries** — When a chaos experiment reveals an unexpected
   failure mode, the agent writes it back to the corpus.

The Chaos Agent queries the Brain for relevant scenarios before each experiment,
prioritizing scenarios similar to the current system-under-test. Scenarios
include a `last_tested` timestamp and a `times_passed` counter to enable
regression testing and retirement of stale scenarios.

- **Pro:** Self-growing corpus; leverages existing agents and the Brain's
  semantic search; covers both adversarial and operational failure modes.
- **Con:** Requires schema for corpus entries; Brain dependency means corpus is
  unavailable if the Brain is down (mitigated by a local cache).

**Why this is best:** It turns every failure — discovered by red teaming,
chaos experiments, or real incidents — into a reusable test case. The Brain's
semantic search means the Chaos Agent doesn't just replay old tests; it finds
*relevant* tests for the current context. This is the only option that
creates a compounding knowledge advantage.

### Option C — LLM-Generated Corpus

Use a Tier 1 model to generate attack scenarios from system architecture
descriptions. The model produces scenarios; a human reviews and approves them
before they enter the corpus.

- **Pro:** Rapid corpus growth; can generate scenarios for systems that haven't
  failed yet.
- **Con:** Generated scenarios may be superficial or redundant; human review
  bottleneck; no grounding in actual system behavior.

## 6. Handoff Protocol JSON Schema

**Current state:** The handoff protocol is structured text with named fields
(FROM, TO, VIA, TASK, etc.). Domain-specific interface contracts add
field-level detail. Governance handoffs already use JSON-like structures. There
is no machine-parseable schema, which makes automated validation and tooling
difficult.

### Option A — Full JSON Schema Migration

Replace the text-based protocol entirely with a JSON schema. All handoffs
become JSON objects validated against the schema at send and receive time.

- **Pro:** Machine-parseable; enables automated validation, routing, and
  analytics; eliminates format ambiguity.
- **Con:** Breaking change; loses the human-readability of the current format;
  LLMs may produce malformed JSON more often than structured text.

### Option B — Dual-Format with Canonical JSON **[Recommended]**

Define a JSON Schema as the canonical machine-readable format. The existing
text format becomes a *rendering* of the JSON — generated automatically for
human review and agent prompts. Hooks validate the JSON; agents see the text.

```json
{
  "$schema": "handoff/v1",
  "from": "API Designer",
  "to": "Backend Implementer",
  "via": "Orchestrator",
  "task": "Implement the /users endpoint per spec",
  "deliverable": "...",
  "acceptance_criteria": ["..."],
  "context_files": ["src/api/users.ts", "docs/api-spec.md"],
  "constraints": { "token_budget_remaining": 45000, "deadline": "session" },
  "assumptions": ["..."],
  "open_questions": ["..."]
}
```

A `PostToolUse` hook on the Orchestrator validates outbound handoffs against the
schema. A `PreToolUse` hook on receiving agents validates inbound handoffs.
Interface contracts become JSON Schema `$ref` extensions for domain-specific
fields.

- **Pro:** Backward-compatible (agents still see readable text); enables
  tooling and validation; governance agents can programmatically audit handoff
  quality; interface contracts become schema extensions.
- **Con:** Two representations to maintain (mitigated: text is auto-generated
  from JSON); slightly more complex handoff pipeline.

**Why this is best:** It gives you machine validation without sacrificing the
human-readable format that LLMs produce reliably. The dual-format approach
means zero disruption to existing agent prompts while enabling the tooling and
analytics the framework needs at scale.

### Option C — Structured Text with Field Validation Hooks

Keep the text format but add hooks that parse and validate specific fields
(e.g., ensure `TO` matches a known agent role, `CONTEXT FILES` exist on disk).

- **Pro:** No format change; incremental improvement.
- **Con:** Fragile parsing of free-text fields; can't support complex
  validation (e.g., schema-level type checking on `constraints`); poor
  foundation for tooling.

## 7. Model API Outage Recovery

**Current state:** The framework enforces model tier assignments via
`SessionStart` hooks and documents multi-model patterns (economy drafting,
cross-model review). There is no documented strategy for handling model API
outages, rate limits, or provider failures.

### Option A — Provider Abstraction Layer with Automatic Failover

Introduce a provider abstraction that maps each model tier to a primary and
fallback provider (e.g., Tier 1: Claude Opus → Claude Sonnet; Tier 2: Claude
Sonnet → Claude Haiku). On API failure (HTTP 5xx or timeout > 30 s), the
abstraction retries once, then fails over to the fallback. A governance alert
fires on every failover.

- **Pro:** Transparent to agents; automatic recovery; tier degradation is
  controlled and logged.
- **Con:** Fallback model may not meet the quality bar for the agent's role
  (e.g., Security Auditor on Haiku); adds a new infrastructure component;
  cross-provider failover requires API compatibility.

### Option B — Tiered Degradation with Quality Gates **[Recommended]**

Define a per-agent degradation policy in the agent's metadata:

```
### API Resilience
- **Primary:** claude-opus-4-6
- **Degraded:** claude-sonnet-4-6 (max 2 tasks before re-evaluation)
- **Blocked:** None — queue tasks for primary recovery
- **Rate Limit Strategy:** Exponential backoff (1s, 2s, 4s, 8s, max 30s)
```

Rules:
1. On API failure, retry with exponential backoff (up to 4 retries).
2. If primary is unavailable after retries, check degradation policy.
3. If `Degraded` is defined, switch to fallback model **but** apply a stricter
   quality gate — the governance layer reviews degraded-model outputs at 2x the
   normal audit rate.
4. If `Blocked`, queue the task and alert the Admiral. No silent degradation.
5. Governance Token Budgeter tracks degraded-mode token spend separately.
6. On primary recovery, flush the queue and resume normal operation.

- **Pro:** Per-agent control (Security Auditor can be `Blocked` while a
  Frontend Agent can degrade to Sonnet); explicit quality gates prevent silent
  quality erosion; no new infrastructure — policies live in agent definitions.
- **Con:** Requires defining degradation policy for every agent; queue
  management adds complexity.

**Why this is best:** Different agents have different quality sensitivity. A
blanket failover policy either under-protects critical agents or over-blocks
non-critical ones. Per-agent degradation policies let you make the right
trade-off for each role, and the increased governance audit rate ensures
degraded outputs don't slip through unchecked. The policy lives in the agent
definition — no new infrastructure needed.

### Option C — Circuit Breaker with Manual Recovery

Implement a fleet-wide circuit breaker. On N consecutive API failures within a
time window, the circuit opens — all new tasks are queued, in-flight tasks
complete with cached context, and the Admiral is notified. A human operator
closes the circuit when the API recovers.

- **Pro:** Simple; prevents cascade failures; explicit human-in-the-loop for
  recovery.
- **Con:** Fleet-wide halt is heavy-handed; minor provider hiccups block all
  work; requires human intervention to resume.

## Priority Order for Implementation

| Priority | Item | Rationale |
|----------|------|-----------|
| **P1** | 7 — Model API Outage Recovery | Fleet is non-functional without the model API. This is the only item with zero mitigation today. |
| **P2** | 1 — Orchestrator Failover | Second-most critical SPOF; governance bypass is a partial mitigation but not a real solution. |
| **P3** | 3 — Governance Self-Monitoring | The immune system must be healthy for everything else to work. |
| **P4** | 6 — Handoff Protocol Schema | Enables automated validation and tooling for all inter-agent communication. |
| **P5** | 5 — Attack Corpus Bootstrapping | Compounding value — the sooner it starts collecting, the more useful it becomes. |
| **P6** | 4 — Hook Configuration Spec | Important for ecosystem growth but current convention works for small hook sets. |
| **P7** | 2 — Identity Authority SPOF | Lowest current risk; the agent is stateless and the real fix is runtime hooks. |
